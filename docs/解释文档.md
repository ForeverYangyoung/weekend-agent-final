# 解释文档：实现逻辑、演示叙事与未来方向

> 用于说明 Demo 里实现了什么、为什么这样演示，以及 Zero-Skill 未来方向；2 页正式交付见根目录 `设计文档.md`。

---

## 1. 我们实现了什么

这是一个 **本地短时活动规划 + 执行 Agent**。用户一句话提出周末出行需求，系统完成：

1. **画像抽取**：Profiler 解析场景、人数、时间、距离、饮食偏好。
2. **隐式档案注入**：用 Demo Mock 模拟跨端历史档案，例如家庭控糖控卡、朋友恢复期禁辣。
3. **候选检索与规划**：Researcher 查 POI，Planner 组合 Top-2 方案。
4. **规则校验与 HIL**：如果历史档案约束和用户显式偏好冲突，不静默替用户决定，而是让用户删标签后重规划。
5. **预检与执行**：DryRun 查票/桌位；用户确认后 Executor 订票/订桌/送附加项。
6. **异常自愈**：满座或失败时，Planner 拉黑问题 POI，换备选店；写操作失败时 Compensator 回滚。

## 2. 统一的叙事逻辑

核心叙事不是“推荐一家店”，而是：

> **显式需求听用户的，隐式档案帮用户兜底；冲突时交给 HIL，而不是系统静默取舍。**

所以家庭线和朋友线现在是同一套逻辑：

| 场景 | 用户显式偏好 | 历史档案 Mock | 冲突处理 |
|------|--------------|---------------|----------|
| 家庭 | 加火锅 / 重口味 | 老婆/家庭控糖控卡 → 低卡、少糖 | 黄条提示：低卡档案 vs 火锅，用户删除一边再规划 |
| 朋友 | 重口味聚餐 | 小明痔疮/上火恢复期 → 禁辣 | 黄条提示：禁辣档案 vs 重口味，用户删除一边再规划 |

这两个例子都不是“多做一个功能”，而是展示同一个能力：**跨端隐式约束进入 Planner 前的画像层**。

---

## 3. 推荐演示顺序

### 主推：朋友线

1. 选择「朋友」场景。
2. 用户输入/预设里只有“4 人、重口味”，没有说禁辣。
3. 右侧 Trace 出现：
   `Zero-Skill·隐式画像(Mock) | 痔疮/上火恢复期 → 禁辣`
4. 左侧出现「禁辣·档案」标签和黄条冲突。
5. 说明：这不是已经接入真实 LTM，而是用 Mock 演示未来美团跨端档案的价值。
6. 继续看姜虎东 4 人满座 → Recovery 换店。

### 备选：家庭线

1. 选择「家庭」场景。
2. 添加「火锅」或「重口味」。
3. 系统保留历史档案的低卡/少糖，不会静默删除。
4. 左侧黄条提示低卡档案与火锅冲突，用户删除低卡或删除火锅后重规划。

## 4. 未来方向：Zero-Skill 隐式画像

本 Demo 只做 **Zero-Skill 的 Mock 切口**：

- 家庭：健康/控糖控卡档案影响饮食约束。
- 朋友：痔疮/上火恢复期影响禁辣约束。

正式版可以接入：

1. 美团买药、外卖、到店偏好等跨端行为。
2. 长期画像 / 向量记忆 / 异步反思 Agent。
3. 自动更新“当前有效约束”，供 Planner 消费。

执行层不需要大改：DryRun、Executor、Compensator 仍然按当前结构工作。

---

## 5. 一句话总结

赛题要求我们“把事情做完”；当前 Demo 证明规划和执行闭环已经跑通。  
Zero-Skill 是产品纵深：让用户不必每次重复说身体状态、家庭偏好和长期习惯，Agent 能在不越权的前提下主动提醒，并通过 HIL 让人做最终选择。
